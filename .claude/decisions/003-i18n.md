# ADR-003 · Bilingual i18n architecture (EN + IT)

**Date**: 2026-05-27
**Status**: Accepted

## Context

The owner writes natively in English and Italian and wants both languages first-class from day one. Per Session #2 discussion:

1. **Strategy**: path-based (one domain, `/en/` and `/it/` prefixes) — chosen over per-domain strategy.
2. **Default fallback locale**: English. Browsers without an Italian preference land in EN.
3. **Slugs**: distinct per locale (e.g., `tube-screamer-circuit.mdx` in EN vs `dentro-il-tube-screamer.mdx` in IT). SEO-pure per language.
4. **Pathname localization**: URL segments are also localized — `/en/posts/foo` vs `/it/articoli/foo`, `/en/tags/foo` vs `/it/tag/foo`.

Constraint: GitHub Pages static export means no server-side middleware, so locale routing must be statically pre-generated and the root `/` redirect must be browser-side.

## Decision

Adopt **`next-intl`** as the i18n library. Configure:

```ts
// src/i18n/routing.ts
{
  locales: ['en', 'it'],
  defaultLocale: 'en',
  localePrefix: 'always',
  pathnames: {
    '/': '/',
    '/posts/[slug]': { en: '/posts/[slug]', it: '/articoli/[slug]' },
    '/tags':          { en: '/tags',          it: '/tag' },
    '/tags/[tag]':    { en: '/tags/[tag]',    it: '/tag/[tag]' },
    '/feed.xml':      '/feed.xml',
  },
}
```

### URL structure

```
/                                  → JS splash, redirect to /en/ or /it/
/en/                               → English home
/en/posts/[slug]/                  → English post page
/en/posts/[slug]/og.png            → built OG image
/en/feed.xml                       → English RSS / Atom / JSON
/it/                               → Italian home
/it/articoli/[slug]/               → Italian post page
/it/articoli/[slug]/og.png         → built OG image
/it/feed.xml                       → Italian RSS / Atom / JSON
```

### Content organisation

```
content/posts/
├── en/
│   └── tube-screamer-circuit.mdx     ← articleId: "tube-screamer"
└── it/
    └── dentro-il-tube-screamer.mdx   ← articleId: "tube-screamer"
```

Each `.mdx` frontmatter carries an `articleId` field (string). Two posts with the same `articleId` are considered translations of one another and are linked by the locale switcher. A post with no `articleId` is treated as locale-exclusive (no link to the other locale; switcher falls back to the other-locale home).

### Root locale redirect

`src/app/page.tsx` (Server Component) renders a minimal HTML splash. It contains:

1. `<meta name="robots" content="noindex">` — splash not indexed.
2. `<link rel="canonical" href="/en/">` — canonical points to default locale for crawlers that ignore meta-refresh.
3. A small inline `<script>` that:
   - Checks `localStorage['preferred-locale']`; if set, redirects immediately.
   - Otherwise reads `navigator.language`; if it starts with `it`, redirects to `/it/`.
   - Otherwise redirects to `/en/` (default).
4. A `<noscript>` block with two manual links (`/en/`, `/it/`).

The script uses `location.replace` (not `assign`) so the splash doesn't pollute the back-button history.

### Locale switcher behaviour

A small `IT / EN` toggle in the header. On click:

1. Writes `localStorage['preferred-locale']` so future visits to root respect the choice.
2. If on a post page and the current post has an `articleId`, computes the gemella's URL in the target locale and navigates there.
3. Otherwise navigates to the home of the target locale.

### Translations of UI strings

Live in `messages/{en,it}.json`. Loaded by `src/i18n/request.ts` via `getRequestConfig`. Used in components via `useTranslations()` (client) or `getTranslations()` (server).

Initial UI string set: ~30 strings (section titles, "Read more", "Continue reading", date prepositions, "min read", "Previous", "Next", "Writings", "About me", footer labels, etc.).

### Date and number formatting

Delegated to the browser's `Intl` API. `next-intl` exposes `useFormatter()` which wraps it with the current locale automatically.

### SEO

`next-intl` injects `<link rel="alternate" hreflang="...">` for each page's translations in the document head. The sitemap (`src/app/sitemap.ts`) iterates both locales and emits `<xhtml:link rel="alternate" hreflang="...">` annotations.

## Consequences

**Positive**:

- Two clean URL trees, each SEO-pure in its language.
- Adding more locales later (FR, ES, etc.) is mostly additive — add to `locales` array, add message file, add content folder.
- Pathname localization gives the URL itself a native feel for native readers.

**Negative / trade-offs**:

- Slug pairing requires the `articleId` frontmatter field. If forgotten, locale switcher falls back to home (graceful, but loses cross-link).
- Every post written in one locale is candidate for translation. Authoring discipline: don't promise bilingual content where you can't deliver it; locale-exclusive posts are explicitly OK.
- The root JS splash adds 100–200 ms latency for visitors landing on `/`. Direct-link visitors (e.g., from a tweet pointing at `/en/posts/...`) bypass it entirely.

## Revisit triggers

Reopen this decision if:

- A third locale is wanted (likely additive but worth checking the message-file management gets unwieldy).
- Pathname localization causes SEO issues unforeseen in the docs.
- `next-intl` becomes unmaintained or a better library emerges.
